🔧 qa_gate.py 關鍵修正點（2026-07-16 實戰新增）

檢查項問題修正
Links (內部連結 HTTP 200)檢查器用 filesystem 路徑檢查 href="/mklab-stock/"，本地無此目錄 → 404解析 href 時移除 /mklab-stock/ 前綴再檢查本地路徑
CSS inline style掃描到 <style>...</style> 區塊內的 CSS 規則（如 .kv .small { font-size:13px; }）誤判為 HTML inline style先用 regex re.sub(r'<style>.*?</style>', '', src, flags=re.S) 移除 style 區塊再檢查
HTML 結構健康檢查<pre><code> 區塊內的程式碼示例（如 <mklab-kline>、<mklab-datatable> 等自訂元素標籤）被 HTML parser 解析為真實標籤，導致誤判「未關閉標籤」增強 StructureChecker 類，將 <pre>、<code>、<textarea> 加入 RAW_CONTENT_TAGS，進入這些標籤內部時完全跳過結構解析（不追蹤嵌套標籤，直接忽略）。測試通過：mklab-stock-log.html 含大量程式碼區塊的檔案現在正確通過檢查。

📦 ETF 資料處理增強（2026-07-16/17 完善）

OHLC 回填與 market_cap 估算

問題：TWSE API 對 ETF 僅回傳 ClosingPrice，無 Open/High/Low；market_cap 為 null。

解決：fetch_data.py 和 export_db.py 同步加入邏輯：

  • OHLC 回填：有收盤價但缺 OHLC 時，用 close 回填 open/high/low（視為平盤）
  • market_cap 估算：靜態 data/etf-shares.json 記錄 ETF 發行張數，以 price * shares_stock 估算
  • 產業別修正：ETF 預設產業別設為 25（金融保險），避免 ind=null 觸發 QA Gate ERROR
  • 兩支線同步修改確保 GitHub Actions (fetch_data.py) 和本機灌種 (export_db.py) 行為一致

ETF 代號分類與 OHLC 處理

尾碼類型說明OHLC 來源
A主動式 ETF經理人主動選股/調倉STOCK_DAY_ALL 有完整 OHLC
R反向型 ETF走勢與標的相反 (-1×)STOCK_DAY_ALL 有完整 OHLC
L槓桿型 ETF放大單日報酬 (2×、3×)STOCK_DAY_ALL 有完整 OHLC
U期貨/美元計價 ETF追蹤商品期貨/匯率部分有 OHLC，部分無
B債券型 ETF固定收益為主視標的而定

QA Gate 髒值檢查決策規則（qa_gate.py 的 無髒值 檢查，改腳本必守）

  • OHLC（open/high/low/price/volume）：全部為 null = 資料源未涵蓋（ETF/字母尾碼標的）→ WARNING（非阻擋）；部分為 null（其他有值）= 腐化/匯出錯誤 → ERROR（阻擋）。這區分「源缺口」與「真髒資料」，不可合一。
  • market_cap：雲端來源（TWSE STOCK_DAY_ALL / BWIBBU）不回傳市值，前端顯示 -，故 market_cap=null 僅 WARNING；若為非數值/負數才 ERROR。切勿把 market_cap 加回 Required-null-ERROR（會對所有 ETF 永久 BLOCK）。
  • pe/pb/div/roe/eps/rank/alert：可為 null（ETF/外股常缺）→ 不計入錯誤。
  • 帶字母尾碼的 ETF（如 00400A、00625K）由 _etf_suffix() 判定為已知缺口，OHLC 缺值降 WARNING。

經驗法則：閘門報 ERROR 時，先判斷是「腐化（部分欄位缺）」還「資料源未涵蓋（全缺/已知 ETF）」。前者必須修資料管線（如 fetch_data.py 對無成交標的設 None、ETF 由 STOCK_DAY_ALL 補 OHLC）；後者若為已知 ETF 缺口則列 WARNING。若修管線成本高，至少報告明確標註「已知缺口」，勿假稱通過。

⏰ Cron Job 輸出簡化模式（2026-07-17）

問題：定時任務輸出長串過程 log，Telegram 推送過長且用戶只關心結果摘要。

解決：為每個 batch 建立 Python wrapper（batch{1,2,3}_wrapper.py），執行原腳本後從 stdout/log.md 提取關鍵統計，只輸出 15-20 行摘要：

  - 寫入價格筆數: 1360
  - 新增股票檔數: 5

📈 覆蓋率:
  - PE/PB 覆蓋: 1078/1360
  - 總資料筆數: 350000 (跨 260 個交易日)
  - 最新交易日: 2026-07-15

📋 股票庫存量:
  - 上市/上櫃股票: 1710 檔
  - ETF: 90 檔
  - 全部股票: 1800 檔

腳本位置：/root/.hermes/scripts/batch{1,2,3}_wrapper.py

關鍵技巧：

  • 用 sqlite3 直接查 stock_overview 取得當下股票/ETF 檔數
  • 從原腳本 stdout 或 log.md 最近一輪 batch 記錄提取統計

經驗法則：閘門報 ERROR 時，先判斷是「腐化（部分欄位缺）」還「資料源未涵蓋（全缺/已知 ETF）」。前者必須修資料管線（如 fetch_data.py 對無成交標的設 None、ETF 由 STOCK_DAY_ALL 補 OHLC）；後者若為已知 ETF 缺口則列 WARNING。若修管線成本高，至少報告明確標註「已知缺口」，勿假稱通過。

🔍 頁面失效的逐步診斷法（getComputedStyle 對比、二分法、console 實測）

本專案三次誤判：

  • watchlist 線上空白 → 以為是 browser 工具 bug，實際是缺 </style> 導致整個 <body> 被 parser 吞掉（HTML 合法但失效）。
  • watchlist 黃標無樣式 → 以為 CSS 與他頁一致，實際是 <style> 漏定義 .freshness/.section-title 規則。
  • 排序「無效」→ 實際是瀏覽器快取舊版。

正確做法：用 browser_console 讀 getComputedStyle(el) 對比已知正常頁（如 industry），並實際 click() 表頭/按鈕測試行為，再判斷。單一 empty snapshot 不足以斷定失效。詳見 references/diagnostic-recipe.md。
